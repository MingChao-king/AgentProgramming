# Dispatching Parallel Agents：并行调度策略

## 技能类型：柔性

决定"何时并行、如何分派、怎么整合"。不替代 subagent-driven-development —— 那是单任务执行质量保证。

---

## 决策矩阵

在分派之前，先回答三个问题：

```
Q1: 任务是否互不依赖？
    ├─ 任务 A 的输出是任务 B 的输入？ → 串行
    └─ 互不依赖 → 继续

Q2: 任务是否操作不同文件？
    ├─ 两个任务改同一个文件？ → 串行（防止合并冲突）
    └─ 不同文件 → 继续

Q3: 任务是否不共享可变状态？
    ├─ 共享数据库/缓存/配置？ → 评估后决定（可能需要串行）
    └─ 不共享 → 可并行
```

### 适合并行

| 场景 | 示例 |
|------|------|
| 新建互不相关的文件 | 同时创建 DTO A、DTO B、DTO C |
| 修改不同模块 | 改 user-service 的 Service + 改 order-service 的 Controller |
| 独立测试文件 | 同时写 4 类测试（隔离/边界/并发/异常） |

### 不适合并行

| 场景 | 原因 |
|------|------|
| 同一文件密集修改 | 合并冲突 |
| 共享数据库 schema 变更 | 可能互相破坏 |
| 根因高度相关的 bug | 独立修复可能互相覆盖 |

## 分派流程

```
1. 从 writing-plans 加载任务清单
       │
2. 按依赖关系分组 → 识别并行组
       │
3. 对每个并行组：
   ├─ 为每个任务构造 5 段式 prompt
   ├─ 同时派发所有 implementer 子代理
   ├─ 等待全部完成
   └─ 每个 implementer 产出进入 3-agent 流水线（spec → quality review）
       │
4. 主代理整合所有通过审查的产出
       │
5. 验证整合结果（编译通过？测试全绿？）
```

---

## 5 段式 Prompt 模板

```
SCOPE: [精确的文件路径 — "src/main/java/.../dto/UserProfileDTO.java"]

GOAL: [可验证目标 — "创建含 nickname/avatar/email 的 DTO，@Data + @Builder"]

CONSTRAINTS:
  1. 写之前搜索项目中是否已有类似代码
  2. 不确定 API 时查 L4 源码
  3. 严格 TDD 三元组（RED → GREEN → REFACTOR）
  4. 只写最少代码（YAGNI）

DO NOT:
  - 不要修改任务范围外的文件
  - 不要"顺便"重构或优化
  - 发现计划外问题 → 汇报，不擅自行动

RETURN: 实现概要 + 测试结果 + 自审结论 + 发现的问题
```

---

## 整合规则

所有子代理完成后：

1. **编译验证**：整合后编译通过
2. **测试验证**：所有测试（含子代理写的）全部通过
3. **冲突检查**：并行修改的文件被 git 识别 → 解决冲突
4. **完整性检查**：对照 writing-plans 任务清单逐项勾销

---

## Red Flags

| 你的想法 | 真相 |
|---------|------|
| "这几个任务可能有依赖，先并行试试" | 不确定依赖关系 = 串行。并行猜错 = 合并冲突。 |
| "虽然是同一个文件，但我可以手动合并" | 同一文件并行修改 = 必定冲突。串行执行。 |
| "整合的时候再检查就行" | 子代理返回后立即对照任务清单逐项验证。事后检查 = 遗漏。 |
| "太多了，全部串行吧" | 独立任务串行 = 浪费 2-3x 时间。门槛是 2。 |

## 完成标准

- [ ] 所有任务已按依赖关系正确分组
- [ ] 每个并行任务已构造 5 段式 prompt 并派发
- [ ] 所有子代理产出已通过 3-agent 流水线审查
- [ ] 整合后编译通过 + 测试全绿
- [ ] 产出对照 writing-plans 任务清单逐项确认

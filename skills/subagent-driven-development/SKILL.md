# Subagent-Driven Development：并行子代理执行

## 技能类型：柔性

并行度和任务分配策略可适配上下文，但子代理必须遵循 AgentProgramming 核心约束。

---

## 何时使用

- 有 3 个以上的独立任务可以并行执行
- 每个任务互不依赖（依赖任务需串行）
- 任务已经在 writing-plans 中明确定义

---

## 执行模式

```
任务清单（来自 writing-plans）
    │
    ▼
为每个独立任务派发一个子代理
    │ → 子代理接收：任务描述 + 文件路径 + 验证方式 + 复用组件列表
    ▼
两级审查
    ├─ 第一级：规格合规（子代理的输出是否符合任务描述？）
    └─ 第二级：代码质量（是否符合项目现有模式？）
```

---

## AgentProgramming 增强：子代理约束

每个子代理在被派发时，必须被告知以下约束：

1. **防重复**：写任何新类/方法前，先搜索项目中是否已有
2. **L4 优先**：不确定 API 签名时，查源码而非凭记忆
3. **L3 利用**：检查项目经验缓存中是否有可复用的组件
4. **最小变更**：只写让任务通过的最少代码（YAGNI）
5. **回报发现**：如果子代理发现了计划外的问题（配置错误、潜在 bug），回报给主代理

---

## 关键行为准则

- 子代理失败时 → 分析失败原因，调整任务描述，重新派发
- 子代理产出偏离计划 → 对比计划和产出，决定是更新计划还是修正产出
- 不要为了"并行"而强行拆分有依赖的任务

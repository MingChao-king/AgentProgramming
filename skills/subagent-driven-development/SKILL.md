# Subagent-Driven Development：并行子代理执行

## 技能类型：刚性

当有 ≥2 个独立任务时，必须使用子代理并行执行。不能以"任务不多"为借口串行。

---

## 何时必须使用

以下条件**全部满足**时，你必须派发子代理：

1. 任务清单中有 ≥2 个任务
2. 这些任务**互不依赖**（任务 A 的输出不是任务 B 的输入）
3. 每个任务在 writing-plans 中已有明确的文件路径和验证方式

**如果条件满足而你不用子代理 = 你在偷懒。**

---

## 执行模式

```
任务清单（来自 writing-plans）
    │
    ▼
为每个独立任务派发一个子代理
    │ → 子代理接收：任务描述 + 文件路径 + 验证方式 + 复用组件列表
    ▼
两级审查
    ├─ 第一级：规格合规（子代理的输出是否符合任务描述？）
    └─ 第二级：代码质量（是否符合项目现有模式？）
```

---

## AgentProgramming 增强：子代理约束

每个子代理在被派发时，必须被告知以下约束：

1. **防重复**：写任何新类/方法前，先搜索项目中是否已有
2. **L4 优先**：不确定 API 签名时，查源码而非凭记忆
3. **L3 利用**：检查项目经验缓存中是否有可复用的组件
4. **最小变更**：只写让任务通过的最少代码（YAGNI）
5. **回报发现**：如果子代理发现了计划外的问题（配置错误、潜在 bug），回报给主代理

---

## 派发流程

### 1. 分组

把任务清单中的任务按依赖关系分组：

```
任务清单：
  #1 创建 UserProfileDTO（无依赖）
  #2 新增 getProfile() 方法（依赖 #1）
  #3 新增 UserAvatarDTO（无依赖）
  #4 新增 /upload-avatar 端点（依赖 #3）

分组结果：
  并行组 A：任务 #1 + 任务 #3（互不依赖，同时派发）
  串行组 B：任务 #2（等 #1 完成）
  串行组 C：任务 #4（等 #3 完成）
```

### 2. 派发

对每个并行组中的任务，使用 `Agent` 工具派发子代理：

```
Agent(
  subagent_type: "general-purpose",
  description: "创建 UserProfileDTO",
  prompt: "
    任务：创建 UserProfileDTO
    文件：src/main/java/.../dto/UserProfileDTO.java
    实现要点：字段 nickname, avatar, email，使用 @Data + @Builder
    验证方式：编译通过 + JSON 序列化包含三个字段

    AgentProgramming 约束：
    1. 写之前先搜索项目中是否已有类似 DTO（Glob **\/dto\/**\/*.java）
    2. 参考已有 DTO 的模式（UserDTO, AdminDTO）
    3. 不确定 Lombok 注解行为时查源码而非凭记忆
    4. 只写任务要求的最少代码
    5. 发现任何问题立即回报
  "
)
```

### 3. 审查

子代理完成后，主代理必须做两级审查：

| 级别 | 审查内容 | 不通过怎么办 |
|------|---------|-------------|
| 规格合规 | 产出是否符合任务描述？文件路径对吗？验证通过吗？ | 修正任务描述，重新派发 |
| 代码质量 | 是否符合项目现有模式？有无重复造轮子？ | 主代理自己修正，或指导子代理修正 |

---

## 关键行为准则

- **子代理失败时** → 分析失败原因，调整任务描述，重新派发
- **子代理产出偏离计划** → 对比计划和产出，决定是更新计划还是修正产出
- **不要为了"并行"而强行拆分有依赖的任务** → 依赖任务必须串行
- **子代理也能用 Agent 工具** → 如果子代理的任务本身又可拆分，可以再派发
- **派发后主代理不要闲着** → 可以去处理有依赖的任务，或准备下一阶段

---

## Red Flags — 你在逃避并行

| 你的想法 | 真相 |
|---------|------|
| "我一个个串行做也很快" | 2 个独立任务串行 = 2x 时间 + 2x token。并行 = 1x。选择很明显。 |
| "就两个任务，不需要子代理" | 2 个独立任务就值得派发。门槛是 2，不是 5。 |
| "子代理也需要上下文，不如我自己做" | 任务描述里已经包含了所有上下文（文件路径 + 验证方式 + 复用组件列表）。 |
| "子代理质量不可控" | 两级审查机制就是为这个设计的。审查不过就重新派发。 |
| "这几个任务太小了" | 小任务也是任务。2 分钟的并行也是并行。 |
| "我先做一个看看，再做第二个" | 这就是串行。如果两个任务真的互不依赖，没有理由不并行。 |

---

## 完成标准

在主代理继续下一阶段之前：

- [ ] 任务清单已按依赖关系分组（识别并行组）
- [ ] 每个并行组已派发子代理（含 AgentProgramming 约束）
- [ ] 每个子代理的产出已通过两级审查
- [ ] 子代理发现的计划外问题已处理
- [ ] 所有并行任务已完成，产出已整合

# Systematic Debugging：四阶段根因分析 + L4 仲裁

## 技能类型：刚性

严格按阶段执行。跳过一个阶段 = 治疗症状而非根除病因。

---

## 四阶段过程

### 阶段 1：复现（Reproduce）

```
→ 找到可靠的重现步骤
→ 确认为什么这个 bug 之前没有被发现
→ 写一个失败的测试来捕获它
```

### 阶段 2：隔离（Isolate）

```
→ 二分法缩小范围：注释掉一半代码 → bug 还在吗？
→ 检查最近的变更（git log / git bisect）
→ 确认是代码 bug 还是环境/配置/数据问题
```

### 阶段 3：根因（Root Cause）

```
→ 不是"修好症状"，而是问 5 次"为什么"
→ ★ L4 为唯一真相源：
    - 检查相关类的实际方法签名（不是你以为的签名）
    - 检查调用链中每一层的实际行为
    - L3 缓存中的踩坑记录是否已有类似问题？
```

### 阶段 4：修复 + 防御（Fix + Defense）

```
→ 先写测试证明修复有效
→ 应用修复 → 确认测试通过
→ 防御：同样的根因在其他地方会发生吗？
    - 搜索类似代码模式（Grep 相同的方法调用模式）
    - 如果找到了 → 一并修复或记录为技术债
→ 回写 L3 踩坑记录（日期 + 问题 + 原因 + 解决 + 严重度）
```

---

## AgentProgramming 增强：L4 仲裁

在调试过程中，任何对代码行为的假设都必须对照 L4 验证：

```
你以为：getUser() 返回 null 时会抛异常
L4 实际：getUser() 返回 Optional<User>，不会抛异常
→ 你的假设错了，bug 在别处
```

---

## 防御深度（Defense in Depth）

修复 bug 后，增加防御层：

1. **输入验证**：在边界处验证输入
2. **错误处理**：给异常路径添加明确的处理逻辑
3. **日志**：在关键路径增加日志（如果之前缺失）
4. **测试**：增加回归测试覆盖这个 bug

---

## Red Flags

| 你的想法 | 真相 |
|---------|------|
| "我觉得问题出在 X" | "觉得"不是调试方法。用二分法隔离，用 L4 确认。 |
| "改一下这行试试" | 随机尝试是浪费时间。每次改动都要有明确的假设和验证。 |
| "这个问题太小了不值得记" | 记入 L3 踩坑记录。同样的坑会发生在下一个 Agent 身上。 |

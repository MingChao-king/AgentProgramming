# Writing Plans：任务规划 + 防重复 + 缓存利用

## 技能类型：柔性

任务拆分的大小和粒度可适配上下文，但防重复检查和 L2/L3 查找是强制的。

---

## 三步工作流

```
设计大纲（来自 brainstorming）
    │
    ▼
第一步：任务拆分
    │ → 产出一份 2-5 分钟/任务的可执行清单
    ▼
第二步：防重复检查（每项任务执行前）
    │ → 确认没有现成的实现可以复用
    ▼
第三步：L2/L3 缓存利用
    │ → 标注可复用的组件、模式、已有类
    ▼
产出：可执行的任务清单（含文件路径 + 验证步骤 + 复用标注）
```

---

## 第一步：任务拆分

### 核心原则

将工作拆分为**极其具体**的任务。每个任务应该是一个热情的初级工程师也能执行的程度 —— 有确切的文件路径、完整的代码逻辑、明确的验证方式。

### 任务粒度标准

| 太粗（不好） | 刚好（好） |
|-------------|-----------|
| "实现用户认证" | "在 `auth/service.py` 中创建 `generate_token(user_id)` 函数" |
| "重构数据库层" | "将 `db/queries.py:45-78` 的 SQL 改为参数化查询" |
| "加个 API 接口" | "在 `api/users.py` 中新增 `GET /users/{id}/profile` 端点，返回 `UserProfileDTO`" |

### 每个任务必须包含

| 字段 | 说明 | 示例 |
|------|------|------|
| **任务描述** | 一句话说清楚要做什么 | "创建 `UserProfileDTO` 响应类" |
| **涉及文件** | 精确到文件路径 | `src/main/java/.../dto/UserProfileDTO.java` |
| **实现要点** | 核心逻辑的关键步骤 | "字段：nickname, avatar, email。使用 @Data + @Builder" |
| **验证方式** | 怎么确认做对了 | "编译通过 + 序列化 JSON 包含三个字段" |
| **依赖项** | 此任务依赖哪个任务先完成 | "需先完成任务 #3（User 实体字段新增）" |

## 第二步：防重复造轮子（强制检查清单）

### ⚠️ 最常见的浪费：写了一个项目中已经存在的东西。

**在规划每个新类/新方法之前，逐项执行以下检查：**

| # | 检查项 | 操作 | 为什么 |
|---|--------|------|--------|
| 1 | 新 Service 方法？ | `Grep "方法名" **/service/**/*.java` | 可能接口已有声明，实现类已有类似方法 |
| 2 | 新 DTO/VO？ | `Glob **/dto/**/*.java`, `Glob **/vo/**/*.java` | 可能已有完全相同的返回结构 |
| 3 | 新工具方法？ | `Grep "方法签名" **/utils/**/*.java`, `Grep "方法签名" **/common/**/*.java` | 项目工具类可能已经封装了 |
| 4 | 新 Mapper/Repository 方法？ | `Grep "方法名" **/mapper/**/*.java`, `Grep "方法名" **/repository/**/*.java` | ORM 内置方法可能已覆盖 |
| 5 | 新常量/枚举？ | `Grep "常量名" **/enums/**`, `Glob **/constants/**` | 避免重复定义 |
| 6 | 新依赖？ | `Grep "artifactId" pom.xml` 或 `build.gradle` | 依赖可能已引入但你不知道 |
| 7 | 功能相似方法？ | `Grep "业务关键词" **/*.java` 模糊搜索 | 换个名字但功能完全一样的方法 |

### 判据：满足以下任一条 → 停止规划新代码，标注"复用已有"

- 已有方法名只差一两个单词，参数列表基本相同
- 已有 DTO 字段覆盖了你需要的 80% 以上（剩余 20% 扩展字段即可）
- 已有工具方法逻辑相同，只是入参类型不同

## 第三步：L2/L3 缓存利用

### L3 项目速查（有什么可以复用）

从 brainstorming 阶段的 L4 勘探结果和 L3 缓存中，标注每个任务可以复用的组件：

```
任务 X：创建 UserProfileDTO
  → 复用：参考已有 DTO 模式（UserDTO, AdminDTO），使用相同的 @Data + @Builder 模式
  → 复用：时间字段复用 BaseEntity 的 createTime/updateTime

任务 Y：实现用户查询接口
  → 复用：UserService.getById() 已有此方法，不需要新写
  → 复用：Result<T> 统一返回格式
```

### L2 跨项目模式（怎么写最优）

在 L2 通用编码经验中搜索当前问题域的最优解：

| 当前任务涉及的场景 | 搜索 L2 的关键词 | 预期命中 |
|-------------------|-----------------|---------|
| 数据分页 | "分页" "Page" "滚动分页" | 滚动分页模式 / MyBatis-Plus 分页 |
| 缓存策略 | "缓存" "Redis" "穿透" | 缓存穿透通用解法 / Redis BitMap |
| 排序保持 | "排序" "ORDER BY FIELD" | MySQL IN 查询保持顺序 |
| 并发控制 | "锁" "乐观锁" "幂等" | 乐观锁 vs 悲观锁选择 |

**L2 命中 → 按模式写代码。L2 未命中 → 放慢速度，精读 L4 源码后谨慎写。任务完成后回写 L2。**

## 任务清单格式

### 最终输出的完整格式

```markdown
## 实施计划：{功能名称}

> 关联设计大纲：[链接]
> 关联 roadmap：{项目}-roadmap.md 阶段 X
> 涉及模块：

### 复用组件汇总

| 已有组件 | 路径 | 用于任务 |
|---------|------|---------|
| UserDTO | .../dto/UserDTO.java | 任务 #1（参考模式） |
| UserService.getById() | .../service/UserService.java | 任务 #3（直接调用） |

### 任务列表

| # | 任务 | 文件 | 复用 | 验证 | 依赖 |
|---|------|------|------|------|------|
| 1 | 创建 UserProfileDTO | `dto/UserProfileDTO.java` | 参考 UserDTO 模式 | 编译 + JSON 序列化 | 无 |
| 2 | 新增 getProfile() 方法 | `service/UserService.java` | — | 单元测试 | #1 |
| 3 | 新增 GET /profile 端点 | `controller/UserController.java` | 调用 #2 | 接口测试 | #2 |
```

## Red Flags — 你在重复造轮子

| 你的想法 | 真相 |
|---------|------|
| "写一个新的更快，搜索已有的太慢" | 写新的 5 分钟 + 调试 20 分钟 + 后续维护。搜索已有 2 分钟。哪个更快？ |
| "这个功能太特殊了，肯定没有现成的" | 你 7 次搜索里有 3 次能找到可复用的。不要靠直觉，靠 Grep。 |
| "已有的那个不太一样，我重写一个更干净" | 扩展已有类比新建类更好。不要为了一个字段新建一个 DTO。 |
| "L2 里没有这个场景的模式" | 那你就有责任在这次任务完成后将其写入 L2。 |
| "就加一个字段，不需要查 L3" | L3 里有这个类的完整字段列表吗？如果超 7 天，需要验证。 |

## 完成标准

在进入实现阶段之前，你必须确认：

- [ ] 所有任务已拆分为 2-5 分钟的可执行单元
- [ ] 每个任务有明确的文件路径、验证方式、依赖关系
- [ ] 防重复 7 项检查已逐项执行
- [ ] 每个任务标注了可复用的已有组件（L3 命中）
- [ ] L2 搜索已完成，相关模式已标注
- [ ] 用户已确认任务清单

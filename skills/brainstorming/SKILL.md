# Brainstorming：需求澄清 + 项目勘探 + 大纲先行

## 技能类型：柔性

将原则适配到具体上下文。但以下三个步骤一个都不能少。

---

## 三步工作流

```
用户提出需求
    │
    ▼
第一步：需求澄清（苏格拉底式提问）
    │ → 产出一份明确的需求文档
    ▼
第二步：L4 项目勘探（了解项目有什么）
    │ → 产出项目版图速查
    ▼
第三步：大纲先行（设计文档 + roadmap 条目）
    │ → 产出可执行的设计大纲
    ▼
进入 writing-plans 阶段
```

---

## 第一步：需求澄清

### 核心原则

**不要假设你理解了用户的意图。** 用提问来提炼模糊的想法，探索替代方案，在投入编码之前确保方向正确。

### 苏格拉底式提问框架

对任何开发需求，至少覆盖以下维度：

| 维度 | 要搞清楚的问题 |
|------|---------------|
| **目标** | 这个功能要解决什么问题？谁会用它？ |
| **范围** | 最小可行版本（MVP）包含什么？什么明确不做？ |
| **约束** | 技术栈限制？性能要求？兼容性要求？ |
| **替代方案** | 有没有更简单的实现方式？现有工具/库能否直接解决？ |
| **验收标准** | 怎么判断做完了？怎么算"正确"？ |
| **测试场景** | 多用户隔离测了吗？权限边界测了吗？空参数/null/极值测了吗？并发冲突测了吗？ |

### 测试场景头脑风暴（必做）

**在需求澄清阶段就必须思考测试场景。实现会限制你的想象力——代码写完了再想测试，你只会测实现能通过的场景。**

对每个需求，至少想出以下四类测试场景：

| 类别 | 要覆盖的场景 | 示例 |
|------|-------------|------|
| **多用户/隔离** | 不同用户/租户的数据隔离？不同角色的权限边界？ | 用户 A 的待办 ≠ 用户 B 的待办 |
| **边界条件** | 空值、null、空列表、极大值、极小值、超长字符串 | userId 为 null 时怎么办？ |
| **并发/竞态** | 同时操作会冲突吗？重复提交会怎样？ | 两人同时编辑同一条记录 |
| **异常路径** | 依赖服务挂了？数据库连接超时？参数非法？ | 文件上传中途取消 |

**每类至少想出 2 条测试场景，写入设计大纲的"测试场景设计"栏。**

### 关键行为准则

- **分块展示设计**：不要一次性倾倒整个设计文档。分 3-5 个可消化的部分呈现，每部分等待用户确认后再继续。
- **主动提出替代方案**：如果你看到更简单或更好的方法，说出来。你的价值不在于执行，而在于判断。
- **明确标注不确定的部分**：如果你对某件事不确定，标注为"待确认"而不是猜测。
- **保存设计文档**：设计大纲保存到知识库的 `knowledge-base/{项目名}/{项目名}-roadmap.md`。**不是**放在项目根目录，**不是**放在 `.claude/`。

---

## 第二步：L4 项目勘探

### 核心原则

**在不知道项目有什么之前，你不知道什么能做、什么值得做。** L4 是唯一真相源 —— 你对项目版图的了解程度直接决定了方案质量。

### 勘探清单

| # | 勘探目标 | 具体操作 |
|---|---------|---------|
| 1 | 项目结构 | 列出所有模块/服务、端口、包结构、职责 |
| 2 | 已有 DTO/实体 | Glob 搜索 `**/dto/**`, `**/entity/**`, `**/vo/**` |
| 3 | 已有工具类 | Grep 搜索 `**/utils/**`, `**/common/**` |
| 4 | 已有接口 | Grep 搜索 `interface.*Service`, `interface.*Mapper` |
| 5 | 代码模式 | 找到 3-5 个类似的现成功能，看它们怎么写（分页？异常处理？返回格式？） |
| 6 | 配置规则 | 配置文件在哪？Nacos？bootstrap？环境变量？ |
| 7 | 依赖清单 | pom.xml/build.gradle 中已有的关键依赖 |

### 勘探输出格式

```markdown
## 项目版图速查

### 模块结构
| 模块 | 端口 | 包名 | 职责 |
|------|------|------|------|

### 可复用的关键类
| 类名 | 路径 | 说明 | 验证状态 |
|------|------|------|----------|

### 现有代码模式
1. [模式名]：[文件路径] — [一句话描述]

### 与本次需求相关的已有组件
- [已有 DTO A]：可复用 / 需扩展字段 X, Y
- [已有 Service B]：已提供方法 Z，不需要重写
```

### L3 缓存核对

在勘探完成后，检查 L3 缓存：

```
知识库中 {项目}-experience.md 是否存在？
  ├─ 存在 → 签名验证前 3 个关键类（方法签名与 L4 一致吗？）
  │          ├─ 一致 → 信任 L3，在编码时使用
  │          └─ 不一致 → 更新 L3 后再用
  ├─ 存在但简陋（<150 行或缺失模块）→ 用 L4 勘探结果补充 L3
  └─ 不存在 → 从 L4 勘探结果生成 L3（任务完成后写入）
```

## 第三步：大纲先行

### 核心原则

**在写任何一行代码之前，必须先有一份设计大纲。** 这不是可选的文档工作 —— 是施工的蓝图。没有蓝图的施工必然返工。

### 设计大纲必须包含

| 要素 | 必填 | 说明 |
|------|------|------|
| **要做什么** | ✅ | 一句话 + 为什么 |
| **模块划分** | ✅ | 新建/修改哪些文件，各自职责 |
| **数据流** | ✅ | 数据从哪来 → 经过什么处理 → 到哪去 |
| **关键决策** | ✅ | 为什么选这个方案（1-2 句话说明放弃的替代方案） |
| **涉及文件** | ✅ | 预估要改/新建的文件清单 |
| **影响范围** | ✅ | 改了会不会影响其他功能？谁在调用会被影响的接口？ |
| **验证方式** | ✅ | 怎么确认改对了？ |
| **测试场景设计** | ✅ | 多用户隔离？权限边界？空值/极值/并发？正常路径+异常路径各至少 3 条 |
| **步骤 Checklist** | ✅ | 可勾选的施工步骤 |

### 新项目 vs 已有项目

**新项目：** 第一步在知识库创建项目文件夹和 roadmap：

```
mkdir -p "${CLAUDE_PLUGIN_ROOT}/knowledge-base/{项目名}"
```

然后按 Roadmap 结构模板（项目目标 → 技术选型 → 阶段规划 → 设计大纲 → checklist）写入 `${CLAUDE_PLUGIN_ROOT}/knowledge-base/{项目名}/{项目名}-roadmap.md`。

**已有项目新增功能：** 在对话中展示设计大纲，获用户确认后追加到知识库中已有的 `${CLAUDE_PLUGIN_ROOT}/knowledge-base/{项目名}/{项目名}-roadmap.md`。

## Red Flags — 你在跳过思考

| 你的想法 | 真相 |
|---------|------|
| "需求已经很清楚了，直接开始吧" | 如果需求真的清楚，走一遍 brainstorming 只需要 2 分钟。如果花了更久，说明需求并不清楚。 |
| "这个项目我已经很熟了，不需要勘探" | 项目在变。你上次勘探是 7 天前，别人可能已经加了新模块。至少验前 3 个关键类。 |
| "就改一行代码，没必要写大纲" | 一行代码改错位置能花 2 小时排查。花 30 秒写 mini 大纲。 |
| "我不确定用什么方案，先写出来试试" | 先想清楚再动手。试错式编码不是探索，是赌博。 |
| "L3 已经有了，我不需要再勘探 L4" | L3 是缓存。超过 7 天必须签名验证。核心业务逻辑无论如何都要验 L4。 |
| "测试场景跑一遍代码就知道了" | 代码写完了再去想测试场景已经晚了——实现会限制你的思维。写代码之前就想好怎么测。 |
| "多用户隔离、权限边界这些是运维的事" | 这些是最常见的功能 Bug。写代码的人不思考隔离 = 注定写出安全漏洞。 |

---

## 完成标准

在进入 writing-plans 之前，你必须确认：

- [ ] 用户需求已明确（目标、范围、约束、验收标准）
- [ ] L4 项目版图已了解（模块/已有类/代码模式/配置）
- [ ] L3 缓存已验证或标注待更新
- [ ] 设计大纲已写出（含模块划分、数据流、关键决策）
- [ ] 用户已确认设计大纲
- [ ] 大纲已保存/追加到知识库 `knowledge-base/{项目名}/{项目名}-roadmap.md`

# Requesting Code Review：对照计划审查

## 技能类型：柔性

审查深度适配上下文（小改动快速过，核心逻辑深入审）。

---

## 审查维度

### 1. 计划合规性

```
对照 writing-plans 的任务清单：
  → 所有任务都完成了吗？有遗漏吗？
  → 产出偏离了计划吗？（有意还是无意？）
```

### 2. 代码质量

```
→ 是否遵循了项目已有代码模式（见 CLAUDE.md § Code Patterns）？
→ 是否引入了重复代码？
→ 命名是否清晰？边界情况是否处理？
```

### 3. L3 契约一致性

```
→ 新增方法签名与 L3 记录一致吗？（有疑问 → 验 L4）
→ 新增接口路由符合项目路由规则吗？
→ 返回格式是项目统一的 Result<T> 吗？
```

### 4. 影响范围

```
→ 修改的接口有哪些调用者？都检查过了吗？
→ 有没有意外的副作用？
```

---

## 问题严重度

| 级别 | 含义 | 行动 |
|------|------|------|
| **严重** | 逻辑错误、安全漏洞、数据丢失风险 | 阻塞合并，必须先修复 |
| **主要** | 偏离计划、遗漏边界处理、性能问题 | 应修复，可在确认后合并 |
| **次要** | 命名不清晰、缺少注释、代码风格 | 建议修复，不阻塞 |
| **建议** | 更好的实现方式、可选的优化 | 可采纳或记录为后续改进 |

---

## 并行审查模式（推荐）

4 个维度天然独立，应**并行派发 reviewer 子代理**（只读），主代理汇总决策：

```
并行派发 4 个 Reviewer（Agent 工具，只读）：

Reviewer A — 计划合规性:
  SCOPE: writing-plans 产出的任务清单 + 本次改动的所有文件
  GOAL: 对照任务清单逐项检查：所有任务都完成了吗？有遗漏吗？产出偏离计划了吗？
  DO NOT: 不要审查代码风格 — 只关注"是否按计划做了"
  RETURN: 逐项勾销结果 + 遗漏/偏离列表

Reviewer B — 代码质量:
  SCOPE: 本次改动的所有源文件
  GOAL: 检查模式一致性（是否遵循 CLAUDE.md § Code Patterns）、重复代码、命名、边界处理
  DO NOT: 不要审查计划合规性 — 只关注"代码写得好不好"
  RETURN: 问题列表（按严重度：严重/主要/次要/建议）

Reviewer C — L3 契约一致性:
  SCOPE: 新增/修改的接口 + L3 experience.md
  GOAL: 方法签名与 L3 一致吗？路由符合项目规则吗？返回格式是统一的 Result<T> 吗？
  DO NOT: 不要重复审查代码质量 — 只关注"契约是否一致"
  RETURN: 不一致项列表（含具体方法签名对比）

Reviewer D — 影响范围:
  SCOPE: 修改的接口 + Grep 搜索结果（谁在调用这些接口）
  GOAL: 列出所有调用者，确认每个调用者不受影响或已同步更新
  DO NOT: 不要审查新代码本身 — 只关注"改了会影响谁"
  RETURN: 调用者清单 + 影响评估
```

**主代理汇总**：4 份报告返回后 → 去重 → 按严重度排序 → 决定哪些必须修复、哪些可记录为后续改进。

**小改动（≤2 文件）**：主代理串行审查即可，并行开销 > 收益。

---

## Red Flags

| 你的想法 | 真相 |
|---------|------|
| "改动很小，快速过一下就行" | 小改动可能有大影响。至少检查调用链。 |
| "代码看起来没问题" | "看起来没问题"不是审查。逐项对照清单。 |

此技能是刚性技能。任何一步都不能跳过。这是 AgentProgramming 与 superpowers 最根本的区别 —— 没有闭环，知识就无法积累，下一次任务仍然是冷启动。
</EXTREMELY-IMPORTANT>

# Closing the Loop：回写闭环

## 技能类型：刚性

严格遵循。不能为图省事而跳过任何步骤。未完成闭环 = 任务未交付。

---

## 核心哲学

**每一次编码不只是完成当前任务，而是为未来所有项目积累资产。**

Superpowers 让每次任务结构化执行。AgentProgramming 更进一步：让每次任务都为下一次加速。

```
没有闭环：每次任务 → 冷启动 → 全量扫描 → 从零开始
有闭环：  每次任务 → 查找缓存 → 命中已有模式/组件 → 快速完成 → 回写新经验
```

---

## 两个仓库 — 什么东西往哪放

**这是最重要的一节。搞错仓库 = 知识库污染或经验丢失。**

```
                      代码仓库                          知识库
                  项目自身的 git remote              gitee.com/gudu-code-man/
                                                  agent-programming-mode.git
                          │                                  │
     ┌────────────────────┼────────────────────┐             │
     ▼                    ▼                    ▼             ▼
  源码             项目级 CLAUDE.md       .gitignore    L1 守则
  (*.java           (项目配置，           (IDE 截图      L2 通用经验
   *.ts             NOT 知识库)           不入仓库)      L3 项目速查
   *.vue                                              └ {项目名}/
   *.xml                                                  ├ {项目名}-roadmap.md
   *.yml                                                  └ {项目名}-experience.md
   ...)
```

| 仓库 | 远程地址 | 存放内容 | 操作时机 |
|------|---------|---------|---------|
| **代码仓库** | 项目自身的 `git remote` | 源码、配置、项目级 CLAUDE.md | 第 1 项：`git commit && git push`（在项目目录执行） |
| **知识库** | `https://gitee.com/gudu-code-man/agent-programming-mode.git` | L1/L2/L3/Roadmap | 第 2 项：`git pull` / 第 5 项：`git add && commit && push`（在 `knowledge-base/` 目录执行） |

### 知识库目录结构

```
${CLAUDE_PLUGIN_ROOT}/knowledge-base/
├── agent编程守则.md              ← L1 方法论（极少改动）
├── 通用编码经验.md                ← L2 跨项目模式库（按技术域归类）
├── Agent四级缓存编程范式.md       ← 元文档（架构总述）
└── {项目名}/                      ← 每个项目一个文件夹
    ├── {项目名}-roadmap.md        ← 项目路线（按时间倒序，分阶段记录每一步）
    └── {项目名}-experience.md     ← L3 项目速查（≥150 行，6 个强制模块）
```

### ⚠ 常见错误 — 改之前先看

| 错误 | 为什么错 | 正确做法 |
|------|---------|---------|
| 把 `{项目名}-roadmap.md` 推到代码仓库 | roadmap 是知识资产，放知识库才能跨项目复用 | 放在 `knowledge-base/{项目名}/{项目名}-roadmap.md` |
| 把 `CLAUDE.md` 推到知识库 | CLAUDE.md 是项目配置，属于代码仓库 | 放在项目根目录（代码仓库） |
| 只在对话里写大纲，不保存到文件 | 下个会话 agent 看不到 | 保存到 `knowledge-base/{项目名}/{项目名}-roadmap.md` |
| 把 L3 experience 写在项目目录里 | L3 是知识库资产，不是代码 | 放在 `knowledge-base/{项目名}/{项目名}-experience.md` |

## 子代理委派回写（推荐）

L2/L3/Roadmap 文件可能非常长（L3 ≥150 行，L2 数万行），全部加载到主对话浪费上下文。**推荐将第 3/4 项和 L2 检查委派给子代理执行**，主对话只做 git 操作。

### 委派流程

```
主代理：准备任务摘要（不读知识库文件）
    │
    ▼
子代理：读取 L3 + roadmap + L2 → 判断更新 → 执行编辑
    │
    ▼
主代理：git pull → git add -A → git commit → git push
```

### 子代理 Prompt 模板

使用 `Agent` 工具派发一个子代理，prompt 如下：

```
你是 closing-the-loop 回写子代理。请完成知识库回写：

**任务摘要**：
- 做了什么：[描述本次代码变更]
- 涉及文件：[文件路径列表]
- 新增/修改的关键类/API/配置：[列表，如无则写"无"]
- 踩到的坑：[描述，如无则写"无"]
- 跨项目经验：[描述，如无则写"无"]

**你的工作**：
1. 读取 knowledge-base/{项目名}/{项目名}-experience.md，判断需要更新什么，执行编辑
2. 读取 knowledge-base/{项目名}/{项目名}-roadmap.md，追加一行实现记录
3. 检查 knowledge-base/通用编码经验.md 是否需要更新（按 L2 判据），如需则执行编辑
4. 回报你修改了哪些文件、改了什么

**注意**：不要执行 git 操作。只做文件编辑。
```

### 主代理后续操作

子代理完成后，主代理只需执行第 2 项和第 5 项的 git 操作（知识库 pull + add + commit + push）。

## 第 1 项：项目代码推送

```bash
cd {项目目录}
git status          # 确认没有遗漏的未跟踪文件
git add -A
git commit -m "{描述本次变更}"
git push
```

**判据**：`git status` 干净，远程仓库有最新 commit。

---

## 第 2 项：知识库拉取最新

```bash
# 首次使用：如果知识库不存在，先 clone
if [ ! -d "${CLAUDE_PLUGIN_ROOT}/knowledge-base/.git" ]; then
  git clone https://gitee.com/gudu-code-man/agent-programming-mode.git "${CLAUDE_PLUGIN_ROOT}/knowledge-base"
fi
cd "${CLAUDE_PLUGIN_ROOT}/knowledge-base"
git pull --ff-only
```

**判据**：输出显示 `Already up to date.` 或成功拉取新内容。

**为什么先拉取**：避免回写时产生冲突。别人（或你自己的另一台机器）可能已经更新了知识库。

---

## 第 3 项：L3 项目速查更新

### 判断 L3 是否需要更新

```
本次任务涉及了哪些新的知识点？
  ├─ 新增了关键类（DTO/Service/Controller/Config）？
  ├─ 新增了 API 接口（含路由规则和签名）？
  ├─ 新增了配置项（Nacos/bootstrap/环境变量）？
  ├─ 发现了新的代码模式（项目中之前没记录过的写法）？
  ├─ 踩到了坑（报错/阻塞/走弯路）？
  └─ 以上任一是 → 更新 L3
```

### L3 质量标准（回写时必须达标）

L3 必须包含 6 个强制模块，不低于 150 行：

| 模块 | 必须内容 | 格式 |
|------|---------|------|
| **项目结构速查** | 所有模块/服务 + 端口 + 包名 + 职责 | 表格 |
| **关键类速查** | ≥10 个核心类，含完整路径和一句话说明 | 表格 |
| **API 契约速查** | 路由规则、鉴权白名单、Feign 接口签名 | 表格 + 代码块 |
| **代码模式** | 项目中 3-5 个最常用的代码写法 | 完整代码块 |
| **配置速查** | 各服务关键配置项 | 表格 + yaml 代码块 |
| **依赖速查** | 关键依赖 artifactId 和用途 | 表格 |
| **踩坑记录** | 每个坑标注严重度(S/M/L)、日期、是否已修复 | 表格 |

### L3 更新操作

```
如果 L3 缺模块 → 不仅加本次发现的内容，补齐所有缺失模块
如果 L3 条目过期 → 更新签名（标注 ✅ 验证日期）
如果 L3 条目不存在于 L4 → 删除或标注"已移除"
```

## 第 4 项：Roadmap 追加

### 每次任务必须在 roadmap 中追加一行

```markdown
| 日期 | 步骤 | 做了什么 | 涉及文件 | 验证结果 |
|------|------|---------|---------|---------|
| 06-19 | 5 | 新增 UserProfileDTO 和用户信息查询接口 | `dto/UserProfileDTO.java`, `controller/UserController.java`, `service/UserService.java` | ✅ 接口测试通过 |
```

### 新项目需创建完整 Roadmap

按以下模板创建 `{项目}-roadmap.md`：

- 项目目标（一句话 + 核心功能描述）
- 技术选型表
- 阶段规划总览
- 当前阶段：设计大纲 + 实现记录 + 测试情况 + 踩坑记录

---

## 第 5 项：知识库推送

```bash
cd "${CLAUDE_PLUGIN_ROOT}/knowledge-base"
git add -A
git commit -m "roadmap: {项目名} - {简短描述本次变更}"
git push
```

**Commit 格式**：`roadmap: {项目名} - {描述}` 例如：`roadmap: hmdp - 新增滚动分页查询`

## 回写规则（第 3/4/5 项的详细操作）

### L2 回写判据（每次任务必查，不是可选项）

**在结束任务前，你必须回答这个问题：本次任务有没有值得其他项目借鉴的经验？**

```
本次任务是否涉及以下任一情况？
  ├─ 发现了一个依赖的版本号陷阱？
  │   （如 jjwt 0.12.6 的 API 与 0.11.5 完全不同）
  ├─ 发现了一个框架配置的反模式？
  │   （如 spring-boot-maven-plugin 不能放在 parent POM 里）
  ├─ 找到了一个跨项目通用的最优解法？
  │   （如 Redis BitMap 签到、滚动分页、SKU 笛卡尔积）
  ├─ 踩到了一个与具体业务无关的技术坑？
  │   （如 MyBatis-Plus 分页插件必须配 PaginationInterceptor）
  ├─ 澄清了一个官方文档的误导？
  │   （如某配置项的实际行为与文档描述不符）
  ├─ 发现了一个通用的性能优化点？
  │   （如 N+1 查询 → 批量查询，冷热分离）
  └─ 以上任一 → 必须回写 L2
       └─ 一个都不沾 → 明确说"本次无 L2 回写"（要有这句话，不能默跳）
```

**L2 回写操作：**

```
打开 knowledge-base/通用编码经验.md
  ├─ 搜索是否已有相关条目（按技术域 grep）
  │   ├─ 已有 → 合并追加到已有条目下（不新建重复条目）
  │   └─ 无 → 按已有条目的格式新建
  └─ 格式要求：
      ├─ 每条必须有 ✅ 推荐做法和 ❌ 避免做法的代码对比
      ├─ 参考字段只能用技术概念（如 "jjwt 0.12.6"），不能用项目名或文件路径
      └─ 标注日期
```

**L2 编写准则**：
- 每条必须是可操作的结论，包含 ✅ 和 ❌ 的对比代码
- 参考字段只能用技术概念（如 "Redis BITFIELD 指令"），不能用项目名或文件路径
- 同问题域已有条目 → 合并追加，不新建

### L3 项目速查更新

见第 3 项的 L3 质量标准和更新操作。

## 冲突优先级（L2 vs L3 给不同方案时）

```
默认：L3（项目已有封装） > L2（通用最佳实践） > 裸写

降级条件（出现任一即降级）：
  L3 与 L4 源码不一致        → L4 为准，立即更新 L3
  L3 记录的版本明显落后       → L2 为准，立即更新 L3
  L2 方案被框架新版本淘汰     → L4 为准，立即更新 L2
```

---

## L3-L4 同步机制

L3 是 L4 的缓存。项目代码在变，L3 必须跟着变。

### 时效性验证

```
打开 L3 → 看"最后更新"日期
  ├─ 距今 ≤ 7 天 → 正常使用
  └─ 距今 > 7 天 → 对将要用的前 3 个关键类做签名验证
       ├─ 签名一致 → 更新日期即可
       └─ 签名不一致 → 标记该条目，更新 L3
```

### 验证状态标记

```markdown
| CartProperties | `...config/CartProperties` | 购物车配置 | ✅ 2026-06-19 |
| XxxService     | `...service/XxxService`    | 某服务     | ⚠️ 2026-06-10 |
| YyyService     | `...service/YyyService`    | 某服务     | ❌ 签名已变   |
```

- ✅ = 已验证，签名与源码一致
- ⚠️ = 超期未验证或有疑点
- ❌ = 已验证，签名已变（必须立即更新）

## Red Flags — 你在跳过闭环

| 你的想法 | 真相 |
|---------|------|
| "这次改动太小了，不值得记" | 最宝贵的经验往往来自小改动（配置项少一个逗号、参数顺序搞反…）。这些小坑不记，下次还会踩。 |
| "太累了，先休息，下次再写 L3" | L3 不写 = 下次冷启动。你现在的"累"换不来下次的"快"。 |
| "这个解法太简单，不算模式" | 如果另一个 Agent 在没有你当前上下文的情况下也能受益于这个信息，就值得写。 |
| "L3 已经很长了，加一条就好" | L3 简陋时 → 不要只加一条，按质量标准补齐所有缺失模块。 |
| "知识库推送可以等下次一起" | 不同机器的 Agent 共享同一个知识库。你不推，别人用旧版本。 |
| "先让用户确认完成，回写是额外工作" | 回写是任务的一部分，不是"额外的"。用户说"完成"的意思是闭环也完成了。 |
| "回写我自己做就行，不用子代理" | L2/L3/Roadmap 加起来可能上百万 token。委派给子代理 = 主对话省下这些上下文给真正的编码任务。不委派 = 浪费。 |

## 完成标准

你必须逐项确认以下所有项：

- [ ] 1. 项目代码已 git commit && git push（推到代码仓库）
- [ ] 2. 知识库已 git pull --ff-only（在 knowledge-base/ 目录执行）
- [ ] 3. L3 experience.md 已检查并更新（文件在 knowledge-base/{项目名}/{项目名}-experience.md）
- [ ] 4. Roadmap 已追加本次实现记录（文件在 knowledge-base/{项目名}/{项目名}-roadmap.md）
- [ ] 5. L2 通用编码经验.md 已检查（有跨项目经验则回写，无则明确说"本次无 L2 回写"）
- [ ] 6. 知识库已 git add -A && git commit -m "roadmap: {项目名} - {描述}" && git push（推到 gitee 知识库）
- [ ] L3 签名验证状态已更新

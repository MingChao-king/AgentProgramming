# Test-Driven Development：TDD + L4 验证

## 技能类型：刚性

严格遵循 RED-GREEN-REFACTOR 循环。不能跳过任何阶段。

---

## RED-GREEN-REFACTOR 循环

```
┌──────────────────────────────────────────┐
│  RED：写一个失败的测试                      │
│    → 测试必须验证一个尚未实现的行为           │
│    → 运行测试 → 确认它失败（红）              │
│    → 如果测试没有失败 → 测试写错了            │
├──────────────────────────────────────────┤
│  GREEN：写最小代码让测试通过                 │
│    → 只写让当前测试通过的代码                 │
│    → 不做任何额外的事情（YAGNI）              │
│    → 运行测试 → 确认它通过（绿）              │
│    → ★ L4 签名验证：使用的关键类方法签名      │
│       与源码一致吗？（3 次工具调用验证）       │
├──────────────────────────────────────────┤
│  REFACTOR：清理代码                        │
│    → 消除重复                               │
│    → 改善命名                               │
│    → 保持测试绿色                            │
│    → 提交（红色 + 绿色 + 重构各一次 commit）   │
└──────────────────────────────────────────┘
```

---

## AgentProgramming 增强：L4 签名验证

在 GREEN 阶段完成后、REFACTOR 之前，验证你使用的关键类：

```
你调用了 UserService.getById(Long id)？
  → 打开 UserService.java → 确认方法签名真的是 getById(Long id)
  → 确认返回类型是 User 还是 Optional<User>
  → 如果 L3 记录不一致 → 以 L4 为准，标注 L3 待更新
```

**为什么要这样做**：LLM 凭记忆写的方法签名有约 15% 的概率是错的（参数名、返回类型、异常声明）。L4 验证将这 15% 的错误在 REFACTOR 之前拦截。

## 测试反模式（参考 superpowers）

| 反模式 | 为什么是反模式 | 正确做法 |
|--------|--------------|---------|
| 测试依赖外部服务 | 不稳定、慢 | Mock 外部依赖 |
| 测试之间有顺序依赖 | 一个失败导致级联失败 | 每个测试独立 |
| 测试方法名不表达意图 | 看不懂测什么 | `should_throw_when_email_is_empty()` |
| 只测 happy path | 80% 的 bug 在边界 | 覆盖 null、空列表、极大值 |
| 为了覆盖率写测试 | 假的安全感 | 测试行为，不测试实现细节 |
| 在生产代码中写 `if (testing)` | 污染代码 | 用依赖注入 |

---

## 关键行为准则

- **在写实现代码之前删除你写的任何代码**（这是 TDD 的铁律）
- **如果测试本身写错了**：先修测试，确认测试变绿不是因为"测错了"
- **不确定 API 签名时**：先查 L4 源码，不要凭记忆写
- **每通过一个测试 → 提交一次**：git commit message 标注 RED/GREEN/REFACTOR

---

## Red Flags

| 你的想法 | 真相 |
|---------|------|
| "这个太简单了，不需要测试" | 简单代码也会有边界 bug。简单 = 测试更简单 = 没有借口。 |
| "先写实现，测试后补" | 后补的测试往往变成"验证实现"而非"验证行为"。TDD 的价值一半在 RED 阶段的设计思考。 |
| "我知道这个方法签名" | 看 L4 确认。15% 的概率你记得是错的。 |

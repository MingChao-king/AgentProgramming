# Verification Before Completion：确认修复 + 冒烟测试

## 技能类型：刚性

在说"完成"之前，你必须主动验证。用户没让你测不等于不需要测。

---

## 冒烟测试清单

| # | 检查项 | 操作 |
|---|--------|------|
| 1 | 编译通过？ | `mvn compile` 或等效命令，零编译错误 |
| 2 | 关键路径能跑通？ | 手动验证核心业务流程（启动 → 调接口 → 看返回） |
| 3 | 边界条件覆盖？ | 空参数、null 值、空列表、极大/极小值各测一次 |
| 4 | 上下游没断？ | 改接口 → 搜谁在调用；改表字段 → 确认关联查询不受影响 |
| 5 | 配置没遗漏？ | 新增配置项 → 确认各环境都有对应值 |
| 6 | 启动日志无异常？ | 检查启动日志，确认没有意料之外的 ERROR/WARN |

---

## AgentProgramming 增强：不相关错误也要指出

在验证过程中，如果发现与当前任务**无关**的问题（编译警告、潜在 bug、配置不一致），不要因为"用户没问"就忽略。

| 情况 | 做法 |
|------|------|
| 发现编译警告 / 弃用 API | 指出并建议修复方案 |
| 发现潜在的空指针 / 资源泄漏 | 指出位置和风险 |
| 发现配置不一致（如端口冲突） | 指出并建议修正 |
| 发现代码逻辑疑似有 Bug | 指出位置和怀疑理由，**不擅自修改** |

**标准格式**：`"另外注意到一个问题：[文件:行号]，[具体问题描述]，建议：[修复方案]。需要我修吗？"`

---

## Red Flags

| 你的想法 | 真相 |
|---------|------|
| "编译过了，应该没问题" | 编译通过 ≠ 逻辑正确。至少走一遍关键路径。 |
| "用户只让我改 X，没让我测 Y" | 冒烟测试是编码的一部分，不是可选的。 |
| "就改了一行，不用测" | 一行改错可以毁掉整个功能。花 30 秒验证。 |
